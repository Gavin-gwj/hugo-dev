Spring Boot 拦截器 (Interceptor) 详解

☕ Spring Boot 拦截器

拦截器 (Interceptor) 是 Spring MVC 框架提供的一种强大机制，允许我们在请求处理的特定阶段（控制器方法执行前后）插入自定义逻辑。它非常适合处理一些通用性任务，例如日志记录、权限检查、身份认证等。

2.5 拦截器 Interceptor

2.5.1 快速入门

什么是拦截器？

  • 它是一种动态拦截方法调用的机制，在概念上与过滤器 (Filter) 类似。
  • 拦截器是 Spring 框架中提供的，专门用于动态拦截控制器 (Controller) 方法的执行。
  • 拦截器的作用：拦截客户端发起的请求，在目标方法调用前后，根据预设的业务逻辑执行相应的代码。

你可以把它想象成控制器方法的“守门员”。

在拦截器中，我们通常会执行一些通用性的操作。例如，我们可以通过拦截器来拦截前端发起的请求，并将登录校验的逻辑统一在拦截器中实现。在校验过程中：

  • 如果用户已登录（例如，携带了合法且有效的 JWT 令牌），拦截器就放行请求，允许其访问 Spring 中的目标资源。
  • 如果用户未登录或令牌非法，拦截器可以直接向前端响应未登录的错误信息，阻止后续访问。

下面我们通过一个快速入门程序来学习拦截器的基本使用。其步骤与过滤器类似，主要分为两步：

  1. 定义拦截器
  2. 注册并配置拦截器

1). 自定义拦截器 (Custom Interceptor)

要创建一个自定义拦截器，你需要实现 org.springframework.web.servlet.HandlerInterceptor 接口，并重写其核心方法。

import org.springframework.stereotype.Component;
import org.springframework.web.servlet.HandlerInterceptor;
import org.springframework.web.servlet.ModelAndView;

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;

// 自定义拦截器
@Component // 将该拦截器声明为 Spring 组件，以便进行依赖注入
public class DemoInterceptor implements HandlerInterceptor {

    /**
     * 目标资源方法执行前执行。
     * @return true：放行；false：不放行，请求将被中断。
     */
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        System.out.println("preHandle .... 拦截器前置处理逻辑执行");
        // 业务逻辑：例如权限检查、日志记录等
        return true; // 返回 true 表示放行，继续执行后续的拦截器和目标方法
                     // 返回 false 表示不放行，请求到此为止
    }

    /**
     * 目标资源方法执行后，视图渲染前执行。
     * 只有当 preHandle 方法返回 true 时才会执行。
     */
    @Override
    public void postHandle(HttpServletRequest request, HttpServletResponse response, Object handler, ModelAndView modelAndView) throws Exception {
        System.out.println("postHandle ... 拦截器后置处理逻辑执行");
        // 业务逻辑：例如修改 ModelAndView 中的数据、添加公共的 Model 属性等
    }

    /**
     * 视图渲染完毕后执行，最后执行的方法。
     * 无论是否发生异常，只要 preHandle 返回 true，此方法总会被调用。
     * 主要用于资源清理工作。
     */
    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        System.out.println("afterCompletion .... 拦截器完成处理逻辑执行");
        // 业务逻辑：例如记录请求处理耗时、异常日志记录等
        if (ex != null) {
            // 处理异常
        }
    }
}

HandlerInterceptor 接口核心方法详解：

  • preHandle(HttpServletRequest request, HttpServletResponse response, Object handler):
    • 在实际的处理器（Controller 方法）执行之前被调用。
    • handler 参数是即将被执行的处理器对象 (通常是 HandlerMethod)。
    • 可以用于进行权限验证、日志记录、请求参数预处理等。
    • 返回值至关重要：
      • true: 执行链继续，后续的拦截器和目标 Controller 方法会被调用。
      • false: 执行链中断。目标 Controller 方法和当前拦截器的 postHandle、afterCompletion 方法都不会被执行。如果返回 false，通常需要拦截器自身来处理响应 (例如，重定向到登录页或返回错误信息)。
  • postHandle(HttpServletRequest request, HttpServletResponse response, Object handler, ModelAndView modelAndView):
    • 在目标 Controller 方法执行之后，但在 DispatcherServlet 渲染视图之前被调用。
    • 可以用于修改 ModelAndView 对象，例如添加额外的模型数据，或者改变视图名称。
    • 仅当 preHandle 方法返回 true 时，此方法才会被调用。
  • afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex):
    • 在整个请求处理完成之后，即 DispatcherServlet 渲染了视图之后执行。
    • 主要用于进行资源清理操作，比如性能监控、日志记录等。
    • ex 参数包含了在处理过程中可能抛出的异常对象。
    • 只要 preHandle 方法返回 true，无论 Controller 方法是否抛出异常，此方法都会被调用。

2). 注册配置拦截器 (Register and Configure the Interceptor)

你需要创建一个配置类 (例如 WebConfig)，实现 WebMvcConfigurer 接口，并重写 addInterceptors 方法来注册你定义的拦截器。

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.InterceptorRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration // 声明这是一个配置类
public class WebConfig implements WebMvcConfigurer {

    // 通过 @Autowired 注入自定义的拦截器 Bean
    @Autowired
    private DemoInterceptor demoInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        // 注册自定义拦截器对象
        registry.addInterceptor(demoInterceptor)
                .addPathPatterns("/**"); // 设置拦截器拦截的请求路径
                                         // "/**" 表示拦截所有请求
    }
}

2.5.2 令牌校验 Interceptor (Token Validation Interceptor)

import lombok.extern.slf4j.Slf4j;
import org.springframework.http.HttpStatus; // 注意：原文使用的可能是 apache.commons.httpclient.HttpStatus
import org.springframework.stereotype.Component;
import org.springframework.util.StringUtils;
import org.springframework.web.servlet.HandlerInterceptor;
// 假设 JwtUtils 是一个用于生成和解析 JWT 的工具类
// import com.example.utils.JwtUtils; 

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;

@Slf4j
@Component
public class TokenInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        // 1. 获取请求 URL
        String url = request.getRequestURL().toString();
        log.info("请求URL: {}", url);

        // 2. 判断请求 URL 中是否包含 "login"，如果包含，说明是登录操作，直接放行
        if (url.contains("login")) {
            log.info("登录请求, 直接放行");
            return true;
        }

        // 3. 获取请求头中的令牌 (token)
        String jwt = request.getHeader("token");
        log.info("请求头中的token: {}", jwt);

        // 4. 判断令牌是否存在，如果不存在，返回错误结果 (未登录)
        if (!StringUtils.hasLength(jwt)) { // 使用 Spring Framework 的 StringUtils
            log.info("获取到 JWT 令牌为空, 返回未授权错误结果");
            response.setStatus(HttpServletResponse.SC_UNAUTHORIZED); // 使用标准 Servlet API 状态码
            // response.getWriter().write("NOT_LOGIN"); // 可以选择返回具体错误信息
            return false;
        }

        // 5. 解析 token，如果解析失败，返回错误结果 (未登录)
        try {
            // JwtUtils.parseJWT(jwt); // 假设这是你的 JWT 解析方法
            log.info("JWT 令牌有效");
        } catch (Exception e) {
            e.printStackTrace();
            log.info("解析 JWT 令牌失败, 返回未授权错误结果", e);
            response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            // response.getWriter().write("INVALID_TOKEN");
            return false;
        }

        // 6. 放行
        log.info("令牌合法, 放行");
        return true;
    }
}

注意: 上述代码中的 JwtUtils.parseJWT(jwt) 是一个占位符，你需要替换为你实际项目中解析 JWT 的方法。同时，HttpStatus.SC_UNAUTHORIZED (来自 Apache Commons HttpClient) 在 Servlet API 中通常使用 HttpServletResponse.SC_UNAUTHORIZED。

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.InterceptorRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Autowired
    private TokenInterceptor tokenInterceptor; // 注入 TokenInterceptor

    // @Autowired
    // private DemoInterceptor demoInterceptor; // 如果还有其他拦截器

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        // 注册 TokenInterceptor
        registry.addInterceptor(tokenInterceptor)
                .addPathPatterns("/**") // 拦截所有请求
                .excludePathPatterns("/login"); // 但排除 /login 请求 (因为登录本身不需要token)

        // 如果有多个拦截器，可以继续添加
        // registry.addInterceptor(demoInterceptor).addPathPatterns("/someOtherPath/**");
    }
}

测试登录校验拦截器 (重要：确保关闭或移除之前的登录校验 Filter)：

  • 测试1：未登录访问受保护页面
    • 关闭浏览器，重新打开。
    • 在地址栏输入需要登录才能访问的页面 URL (例如 http://localhost:8080/depts，假设端口是8080)。
    • 由于用户未登录，TokenInterceptor 会拦截请求，返回未授权状态。前端页面通常会根据这个状态码自动跳转到登录页面。
  • 测试2：先登录，再访问受保护页面
    • 访问登录页面，输入正确的用户名和密码进行登录。
    • 登录成功后，服务器会返回 JWT 令牌，前端通常会将其存储起来 (例如在 LocalStorage 或请求头中)。
    • 再次访问之前受保护的页面。
    • 此时请求头中会携带 JWT 令牌，TokenInterceptor 校验通过后会放行请求，用户可以正常访问相关业务页面。

这验证了 TokenInterceptor 能够正确地实现登录校验功能。在实际项目中，登录校验通常使用 Filter 或 Interceptor 中的一种即可。

2.5.3 Interceptor 详解

了解了拦截器的基本使用后，我们来深入探讨其使用细节，主要包括两个方面：

  1. 拦截器的拦截路径配置
  2. 拦截器的执行流程

2.5.4 拦截路径 (Interceptor Path Configuration)

在注册配置拦截器时，我们需要通过 InterceptorRegistry 的 addPathPatterns("要拦截的路径") 方法来指定拦截器应该拦截哪些请求。

  • addPathPatterns("/**"): 表示拦截所有请求。
  • excludePathPatterns("不拦截的路径"): 用于指定哪些路径不需要被此拦截器拦截。

示例：

@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Autowired
    private DemoInterceptor demoInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(demoInterceptor)
                .addPathPatterns("/**")             // 设置拦截所有请求
                .excludePathPatterns("/login", "/register", "/css/**", "/js/**", "/images/**"); // 设置不拦截的路径，如登录、注册、静态资源
    }
}

常见的拦截路径设置规则 (Path Patterns)：

  • /**: 匹配所有路径。
  • /api/*: 匹配 /api/ 下的一级路径，例如 /api/users，但不匹配 /api/users/profile。
  • /api/**: 匹配 /api/ 下的所有路径，包括多级路径，例如 /api/users 和 /api/users/profile。
  • *.html: 匹配所有以 .html 结尾的路径。
  • /users/{id}: 匹配路径变量，如 /users/123。

原文提到 “暂时无法在飞书文档外展示此内容” 关于常见拦截路径设置的详细列表，但以上是基于 Spring AntPathMatcher 的通用规则，广泛适用。

精确控制拦截与排除的顺序和组合非常重要，以确保拦截器按预期工作。

2.5.5 执行流程 (Execution Flow)

理解拦截器的执行流程有助于我们清晰地认识它与过滤器 (Filter) 以及 Spring MVC 核心组件 (如 DispatcherServlet) 之间的协作关系。

(用户提供的文本中有图片占位符，这里示意一个典型的流程图应该包含的内容)

  1. 客户端请求: 用户通过浏览器或其他客户端发起一个 HTTP 请求到 Web 服务器上部署的 Spring Boot 应用。
  2. 过滤器 (Filter): 如果配置了 Servlet Filter (例如 CharacterEncodingFilter, 自定义 Filter)，请求首先会经过这些 Filter。Filter 会执行其 doFilter 方法中定义的逻辑（放行前逻辑），然后调用 chain.doFilter() 将请求传递下去。
  3. DispatcherServlet: 在 Spring Boot Web 环境中，所有进入 Spring MVC 框架的请求首先由核心的 DispatcherServlet (前端控制器) 接收。Tomcat 等 Servlet 容器本身不直接识别 Spring 的 @Controller，而是通过 DispatcherServlet (它本身是一个 Servlet) 来进行请求的分发和处理。
  4. 拦截器链 (Interceptor Chain) - preHandle:
    • 在 DispatcherServlet 将请求路由到具体的 Controller 方法之前，请求会被注册的拦截器链依次拦截。
    • 每个拦截器的 preHandle() 方法被调用。
    • 如果任何一个拦截器的 preHandle() 返回 false，则请求处理链被中断。后续的拦截器的 preHandle 以及目标 Controller 方法都不会被执行。此时，返回 false 的那个拦截器的 afterCompletion 方法（如果之前有拦截器的 preHandle 返回 true 的话）会被调用，但它的 postHandle 不会。
    • 如果所有拦截器的 preHandle() 都返回 true，则请求继续。
  5. Controller 方法执行: DispatcherServlet 调用目标 Controller 中匹配请求的 Handler 方法。
  6. 拦截器链 (Interceptor Chain) - postHandle:
    • 在 Controller 方法执行完毕，但视图还未渲染之前，拦截器链会逆序执行各个拦截器的 postHandle() 方法。
    • 这意味着最后一个 preHandle() 返回 true 的拦截器，其 postHandle() 会最先被调用。
    • 此阶段可以修改 ModelAndView。
  7. 视图渲染 (View Rendering): DispatcherServlet 根据 Controller 返回的结果 (通常是 ModelAndView 或视图名) 渲染视图。
  8. 拦截器链 (Interceptor Chain) - afterCompletion:
    • 在整个请求处理完成，视图渲染也完毕之后，拦截器链会逆序执行各个拦截器的 afterCompletion() 方法。
    • 此方法用于资源清理。无论处理过程中是否发生异常，只要对应拦截器的 preHandle 返回了 true，它就会被执行。
  9. 过滤器 (Filter) - 放行后逻辑: 请求处理完成后，控制权返回到 Filter 链中，执行 Filter 在 chain.doFilter() 调用之后的逻辑。
  10. 响应客户端: 最终，服务器将 HTTP 响应发送回客户端。

总结 - 过滤器与拦截器的主要区别：

通过执行流程分析，可以清楚地看到过滤器和拦截器之间的主要区别：

  1. 接口规范不同 (Interface Specification):
    • 过滤器 (Filter): 需要实现 javax.servlet.Filter 接口 (Servlet API 的一部分)。
    • 拦截器 (Interceptor): 需要实现 org.springframework.web.servlet.HandlerInterceptor 接口 (Spring MVC 的一部分)。
  2. 拦截范围不同 (Scope of Interception):
    • 过滤器 (Filter): 依赖于 Servlet 容器，它可以拦截几乎所有到达服务器的请求（包括静态资源如 CSS, JS, 图片等），因为它在 DispatcherServlet 之前执行。其拦截范围更广。
    • 拦截器 (Interceptor): 只作用于 Spring MVC 的上下文中，即它仅拦截进入 DispatcherServlet 并由 Spring MVC 处理的请求（通常是动态请求到 Controller 的）。它不能直接拦截静态资源请求 (除非这些静态资源也通过 DispatcherServlet 处理，但这不常见)。拦截器可以访问 Spring 容器中的 Bean。
  3. 使用场景和功能:
    • Filter: 更适合执行一些与 Servlet 规范相关或与具体框架无关的通用任务，如字符编码过滤、XSS 防护、数据压缩等。
    • Interceptor: 更适合执行与 Spring MVC 相关的业务逻辑，如权限检查 (可以方便地获取 Handler 信息)、日志记录 (可以获取即将执行的 Controller 方法信息)、国际化处理等。由于在 Spring上下文中，可以方便地注入和使用其他 Spring Bean。

选择使用 Filter 还是 Interceptor 取决于具体的需求和它们各自的特点。在某些场景下，两者也可能结合使用。
